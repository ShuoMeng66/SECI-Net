# Contributing to SECI-Net

Thanks for helping improve SECI-Net. This guide explains how to propose changes in a way that keeps the project reproducible and easy to review.

## Quick Start

1. Fork the repository and create a feature branch.
2. Make changes with clear commit messages.
3. Run tests or minimal checks when possible.
4. Open a pull request with a short summary, motivation, and any relevant logs.

## What We Appreciate

- Clear, minimal diffs focused on one purpose
- Reproducible training or evaluation steps
- Notes about any data dependencies or expected runtime

## Development Notes

- Keep research artifacts, large checkpoints, and local data out of commits.
- Prefer additive changes with backwards-compatible behavior.
- Update documentation when behavior or flags change.

## Reporting Issues

If you find a bug or want to propose a feature, open an issue with:

- A short title and summary
- Steps to reproduce (or a minimal script)
- Expected vs. actual behavior
- Environment details (OS, Python, PyTorch versions)

## Code of Conduct

Be respectful and constructive in all discussions.
